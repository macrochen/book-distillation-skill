# Book Distillation Skill

## Overview
这个 Skill 不是把书“缩短”，而是把非虚构作品蒸成可迁移的能力。

范围限定：
只处理非虚构类书籍与相关材料，例如投资、商业、管理、写作、传记、思想问答录、演讲整理、访谈实录、方法论文本。

默认不处理：
- 小说
- 散文
- 诗歌
- 戏剧
- 纯虚构叙事文本

核心原则：
不要蒸馏内容，要蒸馏能力。

始终围绕 5 个问题工作：
1. 这本书/这组书在解决什么问题？
2. 它如何观察问题？
3. 它如何下判断？
4. 它如何组织表达？
5. 它在哪些情况下会失效？

目标产物不是普通摘要，而是以下任一结果：
- 一本非虚构书 → 一个专题 Skill
- 同一作者多本非虚构书 → 一个作者操作系统 + 若干子模块
- 同一思想的多版本材料 → 一个版本图谱 + 使用规则
- 一本非虚构书/一组材料 → 可直接复用的判断框架、表达骨架、边界条件

## When to Use
在这些场景使用：
- 用户要把一本非虚构书“蒸馏/提炼/拆解/吸收/转成 Skill”
- 用户要把同一作者的一系列非虚构书蒸成“作者 Skill”
- 用户要比对同一人的同一思想在不同年份、不同问答录、不同整理版本中的稳定内核与演化轨迹
- 用户手里有书摘、章节摘要、读书笔记、原文片段，希望提炼成可复用方法
- 用户要的不是书评，不是摘要，而是“以后遇到新问题还能工作的操作系统”

不要用于：
- 只要剧情梗概、内容简介、作者生平
- 只做金句摘录或书摘归档
- 只写读后感
- 小说、散文、诗歌、戏剧等虚构或文学审美导向文本
- 证据材料过少却要求高置信度作者画像

## Distillation Modes
先分流，不要混做。

### 模式 A：一本非虚构书
产物应是“专题 Skill”。
例如：
- 一本投资书 → 投资判断 Skill
- 一本写作书 → 非虚构写作 Skill
- 一本思维书 → 决策框架 Skill

### 模式 B：同一作者的多本非虚构书
产物应是“作者操作系统”。
推荐拆成：
- Author Core
- Work Skill
- Persona
- Domain Modules
- Corrections

### 模式 C：同一人的同一思想，在不同版本/不同时期书籍中的蒸馏
这不是普通“作者全集蒸馏”，而是“思想版本蒸馏”。
适用于：
- 同一人的问答录有多个年份版本
- 同一主题被多次整理出版
- 同一人的观点先以访谈/帖子出现，后又被编辑成书
- 不同整理者、不同编排方式、不同出版节点导致表达有差异

产物应是：
- Stable Core：稳定不变的思想内核
- Version Map：不同版本/时期各自强调什么
- Drift Map：哪些变化是思想演化，哪些只是编辑/选编差异
- Usage Rules：今天使用时该优先采用哪一版、哪些不能混用

### 模式 D：材料杂糅、尚未整理
先按作者、思想主题或版本谱系拆组，再进入模式 A、B 或 C。
不要直接把不同作者、不同主题、不同目的的材料硬蒸成一个 Skill。

## Non-Negotiable Principles

### 1. 区分“规则”和“例子”
反复出现的判断标准，更像 Skill。
只出现一次的故事，更可能只是例子。

### 2. 区分“主张”和“边界”
没有边界的蒸馏，最后会变成万能废话。

### 3. 区分“作者真正相信什么”和“为了某本书某类读者而这样说”
系列书蒸馏时尤其重要。

### 4. 资料不完整时必须标注覆盖率和置信度
不要把片段印象伪装成完整作者系统。

### 5. 输出要能迁移到新问题
如果产物只能复述原书内容，说明还没蒸成 Skill。

### 6. 对时效性强、准确性要求高的上下文，默认由用户提供
如果蒸馏出的 Skill 在实际使用时，依赖股价、估值、财报、公告、指引、分红、政策口径、行业数据库、实时市场数据或其他强时效信息：
- 默认不能使用 agent 自行搜索得到的时效性数据，来替代用户提供的主口径
- 必须先向用户索取最新、最权威、最认可的信源材料
- 若用户未提供，只能输出框架、待验证问题、低置信度预分析，不能伪装成完整结论
- 若用户提供的数据相互冲突，必须先指出冲突，再请用户确认采用哪组口径

### 7. 蒸馏成 Skill 时，要附“用户提供信息清单”
如果该 Skill 的运行依赖用户提供上下文，尤其是高时效上下文，则最终 Skill 里必须显式加入：
- 需要用户提供哪些信息
- 哪些是必填，哪些是可选
- 每项信息为什么需要
- 缺失后会影响什么判断

优先使用 Markdown 表格表达，方便用户逐项填写。

如果该 Skill 还需要用户把问题表投喂给其他系统辅助整理答案，默认进一步采用“表格展示 + 自动复制到剪贴板”模式：
- 在 Skill 目录中补 `templates/input-checklist.md`
- 在 Skill 目录中补 `scripts/copy_input_checklist.sh`
- 在 Skill 目录中补 `scripts/show_and_copy_input_checklist.sh`
- 在运行到“向用户展示问题表”的阶段时，默认立即执行 `show_and_copy_input_checklist.sh`
- 若复制失败，必须明确告知失败原因和模板路径，不允许假装复制成功

通用参考见：`references/checklist-clipboard-pattern.md`

## Required Inputs
开始前至少明确：
- 任务模式：一本非虚构书 / 同一作者多本非虚构书 / 同一思想多版本蒸馏
- 目标产物：专题 Skill / 作者 Skill / 思想版本图谱 / 判断框架 / 表达骨架
- 材料类型：原文、章节摘要、读书笔记、书摘、访谈、前言后记、问答整理、演讲整理
- 使用场景：写作、投资、商业分析、决策复盘、教学、人物模仿等
- 输出深度：快速版 / 标准版 / 深度版

如果用户没明确，但任务意图明显：
- 一本非虚构书默认输出“专题 Skill 草案”
- 同一作者多本非虚构书默认输出“作者总 Skill + 主题模块草案”
- 同一思想多版本材料默认输出“稳定内核 + 版本差异 + 使用规则”

## Source Priority
优先级从高到低：
1. 原书正文、长篇一手文本
2. 作者自己的序言、访谈、问答、演讲、帖子原文
3. 基于一手材料整理成册的问答录、访谈录、演讲录
4. 高质量章节摘要、结构化读书笔记
5. 他人评论、书评、二手总结

规则：
- 一手材料决定主框架
- 整理本要记录整理者、出版节点、材料来源与删改可能性
- 同一思想的不同版本，先判断差异来自作者本人演化，还是来自选编/节选/重写
- 二手材料只用来补充，不用来定义作者核心信念
- 如果只有二手材料，必须显式写“低覆盖率/低置信度”

## Mixed-Format Extraction Rules
处理电子书材料时，不要假设所有文件都能直接全文提取。

推荐顺序：
1. EPUB / HTML / Markdown：优先全文提取，通常可作为高置信度正文来源。
2. 可搜索 PDF：先用文本提取工具拿全文，再判断是否存在乱码、缺页、广告页污染。
   - 实操上优先先做“可提取性检查”，例如先看页数、再抽样前几页，避免一上来就全量处理。
   - 如果 Python PDF 库不可用，优先回退到系统工具（如 `pdfinfo`、`pdftotext`）批量转成 txt，再进入后续蒸馏流程。
   - 多本 PDF 时，优先统一转成纯文本缓存，再做重复项统计、冲突表和证据锚点提取，效率通常更高。
3. 扫描版 PDF：不要假装拿到了全文；先对目录、前言、自序、关键章节做 OCR 摘录，再显式标注“摘录分析”或“OCR 节选分析”。
4. OCR 质量一般时，先蒸馏稳定框架，不蒸馏精确公式、阈值和逐案例细节。
5. 如果全文特别大（例如整本博客合集、问答总集），不要过度依赖普通全文搜索；可先转 txt 后，用脚本做关键词计数、关键句抽样和上下文抓取，再人工判定哪些值得升级为 Stable Core 或 Evidence Anchors。

格式处理后的置信度建议：
- 全文可读的一手文本：高置信度
- OCR 摘录 + 可辨识目录/前言/关键章节：中置信度
- 只有零散页面、截图、二手摘抄：低置信度

如果多本书的材料质量不一致：
- 用高置信度文本决定 Author Core 和 Work Skill 主框架
- 用中低置信度材料补充术、器、案例和表达偏好
- 在最终 Skill 或报告里单独写一节“来源覆盖与置信度说明”

## Core Extraction Table
无论哪种模式，都尽量整理出这张表：

| 字段 | 要提炼什么 |
| --- | --- |
| 核心命题 | 这一章/这本书/这一组书最重要的一句话 |
| 观察起点 | 作者第一眼先看什么 |
| 关键概念 | 高频概念、概念定义、概念之间的关系 |
| 推理顺序 | 作者通常怎样从问题走到结论 |
| 证据偏好 | 信什么，不信什么 |
| 典型句式 | 常见开头、转折、收尾、强调方式 |
| 反例/例外 | 哪些条件下规则失效 |
| 适用场景 | 这套方法适合处理什么问题 |
| 不适用场景 | 哪些问题不能硬套 |

## Workflow

### Step 1: 先判断蒸馏对象
先问自己：
- 这是“一本非虚构书的方法蒸馏”、“一个作者的操作系统蒸馏”，还是“同一思想的版本蒸馏”？
- 目标是做方法，做人设模仿，还是识别稳定内核与版本差异？
- 材料是连续完整，还是零碎片段？
- 这些版本差异来自作者思想演化，还是来自编辑整理与选编取舍？

如果对象不清，先整理材料边界，再继续。
如果材料属于小说、散文、诗歌等文学创作，默认退出本 Skill。

### Step 2: 建立材料清单
列出：
- 书名/章节/版本/年份
- 材料来源
- 覆盖度
- 可信度
- 是否属于作者本人一手表达
- 是否为整理本/选编本/问答录
- 整理者是谁
- 是否存在删节、重排、改写迹象

目标不是求全，而是知道自己到底基于什么在蒸馏。

### Step 3: 做局部卡片

#### 单本非虚构书：按章节做卡片
每章只抓 5 项：
- 本章主张
- 本章关键概念
- 本章论证链
- 本章例子
- 本章边界/例外

#### 同一作者多本非虚构书：按“每本书”做卡片
每本书至少抓：
- 这本书的主命题
- 它服务于哪个主题模块
- 反复出现的判断框架
- 与其他书一致的部分
- 与其他书冲突或修正的部分

#### 同一思想多版本材料：按“版本单位”做卡片
每个版本至少抓：
- 版本名称/年份/整理者
- 该版本主要强调什么
- 与其他版本相比新增了什么
- 与其他版本相比删弱了什么
- 其中哪些变化更像作者思想变化，哪些更像整理差异

要求：
- 不要大段照抄原文
- 尽量用自己的结构化语言重写
- 一个卡片只服务一个分析单位

### Step 4: 做重复项统计
找“反复出现”的东西：
- 反复出现的关键概念
- 反复出现的判断标准
- 反复出现的论证顺序
- 反复出现的证据偏好
- 反复出现的语气和表达骨架

判断原则：
重复出现的，更像可迁移规则。
只出现一次的，更可能是案例或修辞。

### Step 5: 做冲突表
这是蒸馏能否靠谱的关键步骤。

把冲突写成四列：
- 默认规则
- 例外规则
- 触发例外的条件
- 证据来源

常见冲突：
- 前面讲原则，后面给例外
- 早期更锋利，后期更克制
- 面向大众的书和面向专业读者的书说法不同
- 同一个概念在不同作品里定义变了

原则：
不要消灭冲突，要管理冲突。

### Step 6A: 生成单本书的 6 层蒸馏
如果是一本书，最后必须至少产出这 6 层：

1. 主命题层
- 1 个总命题
- 3 到 5 个子命题

2. 问题框架层
- 作者看问题先看什么
- 哪些变量被当成关键变量

3. 推理顺序层
- 从提问到结论的稳定顺序
- 常见中间步骤和转折动作

4. 证据偏好层
- 什么证据更可信
- 什么证据不能单独支撑结论

5. 表达结构层
- 常见开头方式
- 常见推进方式
- 常见收束方式

6. 边界与失效层
- 适用前提
- 典型失效条件
- 不该乱用的场景

### Step 6B: 生成同一作者多本非虚构书的 5 个模块
如果是同一作者多本书，最后必须至少产出：

1. Author Core
- 世界观
- 对人的基本假设
- 价值排序
- 一贯判断口径

2. Work Skill
- 作者如何拆问题
- 如何取证
- 如何比较选项
- 如何形成结论

3. Persona
- 语气
- 句式偏好
- 常见修辞动作
- 说服/反驳/安抚方式

4. Domain Modules
- 按主题拆子模块，不要全混在一起
- 例如：投资、商业、写作、管理、人性观察

5. Corrections
- 明确哪些后期观点修正了早期观点
- 哪些说法只属于特定语境，不升级为通用规则

### Step 6C: 生成同一思想多版本材料的 4 个产物
如果是同一人的同一思想在不同版本/时期中的蒸馏，最后必须至少产出：

1. Stable Core
- 跨版本稳定不变的核心判断
- 不应被版本差异冲掉的底层原则

2. Version Map
- 每个版本的年份、整理者、材料来源
- 各版本各自强化了什么、弱化了什么
- 哪些版本更像“原始表达”，哪些更像“二次整理”

3. Drift Map
- 哪些变化来自作者本人思想演化
- 哪些变化更可能来自编辑、选编、删节、重新排序
- 哪些差异只属于语气变化，不应上升为原则变化

4. Usage Rules
- 今天使用时优先采用哪一版
- 哪些版本适合拿来理解原意
- 哪些版本适合拿来快速入门
- 哪些版本不能混用为一个统一口径

### Step 7: 写成可复用产物
最终输出应尽量是“能直接拿去用”的结构，而不是分析废稿。

如果最终产物是一个需要外部上下文才能运行的 Skill，尤其是投资、商业分析、政策判断、实时数据解读类 Skill：
- 必须显式写出“先向用户索取最新权威材料”的规则
- 必须显式写出“不要用 agent 默认搜索替代用户提供主口径”的规则
- 必须给出一张“用户提供信息清单表格”
- 若用户需要把问题表发给其他系统，默认再补模板文件与 show-and-copy 脚本

推荐输出之一：
- 专题 Skill 草案
- 作者 Skill 草案
- 思想版本图谱
- 任务型提示模板
- 结构化分析框架
- 写作/判断/复盘模板

## Quality Gate
完成前逐项检查：
- 是否蒸出了“方法”而不只是“内容摘要”？
- 是否区分了规则、例子、反例、边界？
- 是否明确了材料覆盖率和置信度？
- 是否处理了前后冲突，而不是粗暴合并？
- 是否能把这套框架迁移到一个新问题？
- 是否避免把作者语气模仿误当成作者判断系统？

如果其中任一答案是否定的，不算蒸馏完成。

## Common Mistakes

### 错误 1：只做摘要，不做规则
修正：必须提炼观察起点、推理顺序、证据偏好和边界。

### 错误 2：只学语气，不学判断
修正：Persona 只能建立在 Work Skill 之后。

### 错误 3：把例子当规律
修正：先做重复项统计，再决定是否升级为规则。

### 错误 4：忽略前后冲突
修正：必须保留默认规则、例外规则、触发条件。

### 错误 5：忽略时期变化
修正：系列书必须标明早期/中期/后期的演化差异。

### 错误 6：把整本书原文硬塞进 Skill
修正：长材料放 references；常驻 Skill 的应是流程、规则、模板、边界。

### 错误 7：把整理版本差异误判成作者思想变化
修正：必须记录年份、整理者、材料来源与删改可能性，再判断变化来源。

### 错误 8：把虚构文学也纳入同一蒸馏框架
修正：本 Skill 只处理非虚构类书籍；文学文本属于另一类分析任务。

## Quick Reference

### 快速判断：你要做哪一种？
- 一本非虚构书、目标是专题方法 → 单本书蒸馏
- 同一作者多本非虚构书、目标是作者判断系统 → 系列书蒸馏
- 同一人的同一思想有多个时期/多个版本/多个整理本 → 思想版本蒸馏
- 材料很碎、作者和主题都混杂 → 先分组再蒸馏

### 一句话公式
单本非虚构书：
章节卡片化 → 找重复框架 → 合并冲突 → 提炼方法 → 写成专题 Skill

同一作者多本非虚构书：
逐本拆解 → 找不变量 → 按主题分模块 → 处理演化与冲突 → 形成作者总 Skill + 子模块

同一思想多版本：
建立版本谱系 → 比对稳定内核 → 区分思想演化与编辑差异 → 生成版本图谱与使用规则

## Deliverable Style
默认输出风格：
- 先讲结论，再讲依据
- 用结构化标题，不堆形容词
- 明确“高置信度 / 低置信度”
- 明确“默认规则 / 例外规则”
- 明确“适用 / 不适用”

如果用户进一步要求真正落地成一个新 Skill：
- 先给蒸馏结果
- 再把结果重写成对应 Skill 的 SKILL.md 骨架
- 必要时补 templates/、references/、README.md
- 若 Skill 依赖时效性强的上下文，默认再补一节“用户提供信息清单表格”和“缺失信息时如何降置信度/不下结论”的规则
- 若 Skill 依赖用户填写表格，且该表格需要复用到外部系统，默认补 `templates/` 与 `scripts/`，并采用 `references/checklist-clipboard-pattern.md` 中的模式
